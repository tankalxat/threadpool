# Custom ThreadPool

Собственная реализация пула потоков с настраиваемым управлением очередями, балансировкой задач и политикой отказа.

Курсовая работа по дисциплине «Многопоточное и асинхронное программирование на Java».

---

## Требования

- Java 21+
- Maven 3.8+

## Сборка и запуск

```bash
# Сборка
mvn clean compile

# Запуск демо-программы
mvn exec:java -Dexec.mainClass="com.pool.demo.Main"

# Запуск конкретного сценария (по номеру 1-8)
mvn exec:java -Dexec.mainClass="com.pool.demo.Main" -Dexec.args="3"

# Запуск бенчмарка
mvn compile && java -cp target/classes com.pool.demo.Benchmark
```

## Архитектура

```
CustomExecutor (интерфейс)
  └── CustomThreadPool
        ├── PoolConfig           — параметры пула
        ├── CustomThreadFactory   — создание именованных потоков
        ├── TaskDistributor       — балансировка задач
        │     ├── RoundRobinDistributor
        │     └── LeastLoadedDistributor
        ├── RejectionPolicy       — политика отказа
        │     ├── AbortPolicy
        │     ├── CallerRunsPolicy
        │     ├── DiscardPolicy
        │     └── DiscardOldestPolicy
        └── Worker                — рабочий поток с очередью
```

### Параметры пула

| Параметр          | Описание                             | Значение в демо |
|-------------------|--------------------------------------|-----------------|
| `corePoolSize`    | Базовое количество потоков           | 2               |
| `maxPoolSize`     | Максимум потоков                     | 4               |
| `keepAliveTime`   | Время простоя до завершения потока   | 5 сек           |
| `queueSize`       | Макс. размер очереди на каждый поток | 5               |
| `minSpareThreads` | Минимум свободных потоков            | 1               |

### Распределение задач

Реализованы два алгоритма балансировки:

- **Round Robin** — задачи распределяются по очередям циклически. Простой и предсказуемый, подходит для однородных
  задач.
- **Least Loaded** — задача направляется в очередь с наименьшим количеством элементов. Лучше при неравномерной
  нагрузке (задачи разной длительности).

По умолчанию используется Round Robin. Переключение через `PoolConfig.Builder`:

```java
PoolConfig config = PoolConfig.builder()
        .corePoolSize(2)
        .maxPoolSize(4)
        .queueSize(5)
        .keepAliveTime(5, TimeUnit.SECONDS)
        .minSpareThreads(1)
        .distributor(new LeastLoadedDistributor())
        .build();
```

### Политики отказа

| Политика                     | Поведение                              |
|------------------------------|----------------------------------------|
| `AbortPolicy` (по умолчанию) | Бросает `RejectedExecutionException`   |
| `CallerRunsPolicy`           | Выполняет задачу в потоке вызывающего  |
| `DiscardPolicy`              | Молча отбрасывает задачу               |
| `DiscardOldestPolicy`        | Удаляет самую старую задачу из очереди |

**Обоснование выбора `AbortPolicy` по умолчанию:** явный отказ сразу сигнализирует о проблеме. В серверном приложении
лучше вернуть ошибку клиенту, чем молча потерять запрос или заблокировать вызывающий поток. Недостаток — требует
обработки исключения на стороне вызывающего кода.

`CallerRunsPolicy` подходит, когда допустима деградация: producer замедляется, создавая естественный backpressure.
Недостаток — блокирует вызывающий поток.

## Демо-сценарии

Класс `Main` содержит 7 сценариев:

| # | Сценарий          | Что демонстрирует                                                                        |
|---|-------------------|------------------------------------------------------------------------------------------|
| 1 | Базовая работа    | 5 задач, все выполнены штатно                                                            |
| 2 | Масштабирование   | 15 задач → пул расширяется до `maxPoolSize`                                              |
| 3 | Idle timeout      | После завершения задач лишние потоки завершаются                                         |
| 4 | Переполнение      | Задач больше ёмкости → все 4 политики отказа (Abort, CallerRuns, Discard, DiscardOldest) |
| 5 | Shutdown          | Новые задачи отклоняются, текущие доработались                                           |
| 6 | minSpareThreads   | Автоматическое создание резервных потоков                                                |
| 7 | CompletableFuture | Пул как `Executor` для `CompletableFuture.supplyAsync()`                                 |
| 8 | shutdownNow       | Немедленная остановка, прерывание потоков, возврат невыполненных задач                   |

Запуск всех сценариев последовательно — без аргументов. Запуск одного — передать номер.

## Анализ производительности

Бенчмарк: отправка N задач (spin-wait ~10 мкс, CPU-bound) с замером общего времени и throughput.

Параметры: `corePoolSize=4, maxPoolSize=8, queueSize=100`. Среда: 12 CPU cores, OpenJDK 21.

| Реализация                     | 1 000 задач | 5 000 задач | 10 000 задач | Throughput    |
|--------------------------------|-------------|-------------|--------------|---------------|
| **Custom Pool (Round Robin)**  | 5 мс        | 24 мс       | 23 мс        | ~434 783 з/с  |
| **Custom Pool (Least Loaded)** | 3 мс        | 14 мс       | 13 мс        | ~769 231 з/с  |
| `ThreadPoolExecutor`           | 7 мс        | 8 мс        | 14 мс        | ~714 286 з/с  |
| `ForkJoinPool`                 | 3 мс        | 14 мс       | 27 мс        | ~370 370 з/с  |

> Запуск бенчмарка: `mvn compile && java -cp target/classes com.pool.demo.Benchmark`

**Анализ результатов:**

- Custom Pool с Least Loaded балансировкой показывает лучший результат (~769K з/с), превосходя `ThreadPoolExecutor` на 8%
  и `ForkJoinPool` вдвое.
- Least Loaded эффективнее Round Robin в ~1.8x за счёт более равномерного распределения задач по очередям.
- `ThreadPoolExecutor` стабилен благодаря единой общей очереди, но при росте числа задач это становится точкой
  конкуренции (contention point).
- `ForkJoinPool` проигрывает на мелких CPU-bound задачах из-за накладных расходов work-stealing механизма.

**Сравнение с Tomcat/Jetty**: оба используют пул потоков с единой общей очередью. Наша реализация отличается привязкой
отдельной очереди к каждому воркеру, что снижает конкуренцию за блокировку очереди при высокой нагрузке, но может
привести к неравномерной загрузке без алгоритма Least Loaded.

### Исследование параметров

| corePoolSize | maxPoolSize | queueSize | keepAliveTime | Время (10K задач) | Throughput  |
|--------------|-------------|-----------|---------------|-------------------|-------------|
| 1            | 2           | 50        | 5s            | 241 мс            | ~41 494 з/с |
| 2            | 4           | 50        | 5s            | 194 мс            | ~51 546 з/с |
| 4            | 8           | 50        | 5s            | 142 мс            | ~70 423 з/с |
| 4            | 8           | 100       | 5s            | 127 мс            | ~78 740 з/с |
| 4            | 8           | 500       | 5s            | 140 мс            | ~71 429 з/с |
| 8            | 16          | 100       | 5s            | 144 мс            | ~69 444 з/с |

**Выводы:**

- Для **CPU-bound** задач оптимально `corePoolSize = availableProcessors`, увеличение `maxPoolSize` не даёт прироста.
- Для **I/O-bound** задач выгоден больший `maxPoolSize` (2-4x от core), т.к. потоки часто блокированы.
- Маленький `queueSize` снижает latency (задачи не ждут долго), но увеличивает число отказов.
- `keepAliveTime` 5-30 сек - компромисс между экономией ресурсов и стоимостью создания потоков.
